# Langfuse is used for LLM observability

All LLM call traces — token counts, latency, cost estimates, and per-session conversation grouping — are captured via Langfuse using LiteLLM's built-in callback system. We chose Langfuse over Helicone because Helicone operates as a proxy: all LLM requests route through Helicone's servers before reaching the provider. This is the same architectural pattern that led us to reject OpenRouter in ADR-0005 — it routes student interaction data through a third-party infrastructure, creating unnecessary exposure under India's DPDP Act given that users are minors. Langfuse uses a callback model: LiteLLM calls the provider directly (as today), then asynchronously forwards trace metadata to Langfuse after the response completes. Student content never transits Langfuse's servers unless prompt logging is explicitly enabled (it is not). The secondary factor is that Langfuse's trace/session model maps naturally to ExplanationSession's multi-turn structure — one trace per session, turns grouped beneath it. Helicone is more request-oriented and less suited to conversation-level analysis. Langfuse is open source and self-hostable, so if cloud data residency becomes a hard requirement it can run inside the same Docker Compose stack with no application code change.

## Considered Options

- **Helicone**: Good dashboard, LiteLLM integration, request-level analytics — but operates as a proxy, routing all LLM traffic through Helicone's servers. Same data-exposure concern as OpenRouter (ADR-0005).
- **Custom `LLMUsageLogger`**: Full control, data stays in our Postgres DB, no third-party dependency — but requires building and maintaining token counting, cost estimation, and a reporting UI ourselves.
- **Langfuse** ✅: Callback model (not a proxy), open source, self-hostable, built-in session/trace grouping suited to multi-turn ExplanationSessions. Observability data stays within our infrastructure in transit; only trace metadata is forwarded asynchronously.
